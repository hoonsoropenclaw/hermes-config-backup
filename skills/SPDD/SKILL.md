# SPDD — Software Process: Research → Spec → Development

A three-phase structured development methodology. Each phase has a dedicated role with strict rules. Complete one phase before moving to the next.

## When to Use This Skill

- Building new features or systems from scratch.
- When you need a disciplined, phase-gated development process.
- When requirements need to be researched before planning, and planning before implementation.
- Use the phases in order: Research → Spec → Implementation.

## Phase Structure

### Phase 1: Research Agent

**File**: `1-research.md` (loaded first)

**Role**: Codebase Research Agent — document and explain the codebase as it exists today. NO suggestions, NO improvements, NO architectural changes. Only describe what exists, where it exists, and how components interact.

**Rules**:
- DO NOT suggest improvements, refactorings, or architectural changes.
- DO NOT perform root cause analysis or propose future improvements.
- ONLY describe what exists, where it exists, and how components interact.
- You are a technical cartographer mapping the current system.

**Steps**:
1. Read the user's request files completely.
2. Decompose the question into research areas (e.g., Routes, Database, UI).
3. Locate where files and components live.
4. Analyze HOW the current code works (without criticism).
5. Find examples of existing patterns for reference.

**Output**:
- Generate `docs/prds/prd_current_task.md` with YAML frontmatter (date, topic, tags, status).
- End with: "Pesquisa concluída. Por favor, dê um `/clear` e carregue `.agente/2-spec.md` para o planejamento."

---

### Phase 2: Spec Agent

**File**: `2-spec.md` (loaded after research phase)

**Role**: Implementation Planning Agent — create detailed implementation plans and be skeptical of vague requirements.

**Rules**:
- Do not write the plan all at once; validate the structure of phases with the user.
- Every technical decision must be made before finalizing the plan.
- The plan must be actionable and complete, without "open questions".

**Steps**:
1. Read `docs/prds/prd_current_task.md` generated by the Research phase.
2. Divide work into incremental, testable phases.
3. For each affected file, define:
   - **Exact path.**
   - **Action:** (CREATE | MODIFY | DELETE).
   - **Logic:** Pseudocode snippets or implementation references.
4. Define "Automated Verification" (scripts/tests) and "Manual Verification" (UI/UX).

**Output**:
- Generate `docs/specs/spec_current_task.md` following the phase template.
- End with: "Spec finalizada. Por favor, dê um `/clear` e carregue `.agente/3-implementation.md` para execução."

---

### Phase 3: Implementation Agent

**File**: `3-implementation.md` (loaded after spec phase)

**Role**: Implementation Execution Agent — implement an approved technical plan with surgical precision.

**Rules**:
- Follow the plan's intent while adapting to reality encountered.
- Implement one phase COMPLETELY before moving to the next.
- **STOP & THINK:** If you find an error in the Spec or a mismatch in the code, STOP and report. Do not guess.

**Steps**:
1. Sanity Check: Read the Spec and the original Ticket. Verify the environment is clean.
2. Execution: Code following Clean Code patterns and Spec snippets.
3. Verification:
   - After each phase, run the "Automated Verification" commands described in the Spec.
   - PAUSE for manual user confirmation after each phase completed.
4. Progress: Update checkboxes (`- [x]`) in the Spec file as you advance.

**Output**:
- Source code implemented.
- Phase completion report with test results.
- **Final Action:** Ask if the user wants regression tests or to proceed to the next task.

## Workflow Flow

```
User Request → [Research Agent] → prd_current_task.md
                                        ↓
                              [Spec Agent] → spec_current_task.md
                                                        ↓
                                              [Implementation Agent] → Code
```

## Key Principles

- **No skipping phases**: Always research before spec, spec before implementation.
- **No guessing**: If something is unclear or mismatched, stop and report.
- **Verification at each step**: Tests and manual checks gate progression.
- **Immutable outputs**: Each phase produces a fixed artifact that gates the next.
