## 1. MVP 範圍宣告

本 MVP 範圍直接採用 consumer-researcher 報告 §4.1 的 9 個 Must 功能,**不做任何 MoSCoW 重新分類**。理由:consumer-researcher 已經把「標籤 OR/AND 篩選、各處室獨立登入、附件上傳、公告 CRUD」依使用者原意列為 Must,本棒只負責把這些 Must 落實成可驗收的產品規格。

### 1.1 MVP 範圍 (Must, 9 項)

| 編號 | 功能 | 主要對應 Persona | 對應標竿 |
|------|------|------------------|----------|
| M-01 | 公告 CRUD (新增、查詢、編輯、刪除) | 小美 | 全部標竿 |
| M-02 | 附件上傳 (PDF / Word / Excel / 圖片) | 小美、佐藤、陳媽媽 | Cloud School、高雄校園通 |
| M-03 | 多標籤 (一則公告可掛多個標籤:年級、班級、處室、活動類型、身分) | 小美、佐藤、陳媽媽 | Notion multi-select、Cloud School |
| M-04 | 標籤 OR/AND 篩選 (含巢狀 group、NOT 排除) | 佐藤、陳媽媽 | Notion Nested Filters、Gmail Label、Jira JQL |
| M-05 | 各處室獨立登入 (教務 / 學務 / 總務 / 輔導 / 校長室各自帳號 + 各自後台) | 小美 | WordPress Roles、Notion Teamspace、Slack Workspaces |
| M-06 | 角色權限分離 (系統管理員 / 處室承辦 / 教師 / 家長 / 學生 / 訪客 至少 5 層) | 全部 | WordPress 6 角色、Notion 6 層、Slack 5 角色 |
| M-07 | 已讀 / 已簽追蹤 (區分「點開」與「確認簽收」) | 小美、陳媽媽 | 高雄校園通雙軌制 |
| M-08 | 推播通知 (公告發布即時通知訂閱者) | 全部 | 全部企業標竿 |
| M-09 | 行動裝置原生體驗 (RWD 網頁或 APP,iOS + Android 都要) | 佐藤、陳媽媽 | 全部台灣標竿都是 APP 為主 |

### 1.2 明確排除 (從 consumer-researcher 報告 §4.4 Won't 與 §1 必抓清單推導)

- 金流整合、線上繳費
- 點名、出缺勤、成績單、課表排課
- 跨校訊息轉發
- 草稿送審排程 (屬於 v2 Should)
- 附件版本歷史、附件預覽 (屬於 v2 Should)
- 多子女切換 (屬於 v2 Should)
- AI 智慧員工 (屬於 v3 Could)
- 到離校刷卡推播 (屬於 v3 Could)

### 1.3 MVP 成功標準 (驗收用)

1. 3 個 Persona 各自至少能完成 3 個典型情境 (見 §2 User Story)
2. 9 個 Must 功能 100% 通過驗收
3. 標籤 OR/AND 篩選必須支援巢狀 group (Notion 2020-05 之後的標準)
4. 行動裝置 (iOS Safari + Android Chrome 最新版) 操作流暢度等同桌機
5. 已讀 / 已簽雙軌制能區分兩種狀態 (對應高雄校園通設計)

## 2. 3 個 Persona × User Story

每個 User Story 採「身為 X,我想要 Y,以便 Z」格式,並附 Given / When / Then 驗收條件。

### 2.1 小美 (教務處行政人員)

#### US-1.1 多標籤 + 多附件公告發布
- **身為** 小美
- **我想要** 在 5 分鐘內發出一則「指定年級 (高三) + 指定身分 (全體學生) + 指定活動類型 (模擬考) + 多個附件 (新考場座位表 PDF、考生須知 Word) + 設定簽收」的多標籤公告
- **以便** 一次發布取代在 email、Line 群、學校官網重複貼文

**驗收條件 (Given / When / Then)**
- **Given** 小美已登入教務處帳號、位於「新增公告」頁面
- **When** 她輸入標題、內文、從標籤選擇器勾選 [高三] + [全體學生] + [模擬考]、上傳 2 個附件 (PDF + Word)、設定簽收截止日為發布後 7 天
- **Then** 公告在 30 秒內送出、立即出現在高三全體學生的推播清單、附件可下載、簽收狀態可在後台追蹤

#### US-1.2 受眾分流:同步給學生 + 家長 + 老師
- **身為** 小美
- **我想要** 在同一則公告中,同時指定「學生」、「家長」、「校內老師」三種不同受眾
- **以便** 重要公告 (例:高三模擬考同意書) 一次送到三種身分的成員,不必分三則發

**驗收條件**
- **Given** 小美正在編輯一則公告
- **When** 她在受眾設定中勾選 [學生] + [家長] + [教師] 三個身分標籤
- **Then** 這三種身分的成員都能在自己的公告列表看到此則、家長與教師的附件權限跟學生相同、已讀 / 已簽狀態按身分分開統計

#### US-1.3 已讀 / 已簽追蹤
- **身為** 小美
- **我想要** 在公告後台看到「目前有幾位家長已讀、幾位已簽、哪些人尚未讀」
- **以便** 對未讀者主動提醒 (打電話或 Line),並對家長「沒收到」的爭議有查證依據

**驗收條件**
- **Given** 小美發出一則需簽收的公告 (例:模擬考同意書) 給高三全體家長
- **When** 她進入公告後台的「簽收狀態」分頁
- **Then** 看到三個數字:已讀人數 / 已簽人數 / 未讀人數,並可匯出完整名單 (含姓名、讀取時間、簽收時間)

#### US-1.4 公告 CRUD
- **身為** 小美
- **我想要** 對我發布的公告做新增、查詢、編輯、刪除
- **以便** 修正錯誤 (例:打錯時間)、更新資訊 (例:附件換新版)、撤回誤發公告

**驗收條件**
- **Given** 小美已登入教務處帳號
- **When** 她進入「我的公告」列表,可看到自己發過的所有公告、按時間排序
- **Then** 點任一則可進入編輯頁、修改後存檔生效;刪除時需二次確認、被刪公告從學生 / 家長列表消失但保留後台稽核紀錄

#### US-1.5 各處室獨立登入
- **身為** 小美 (教務處)
- **我想要** 用「教務處專屬帳號」登入,看不到學務處、總務處的內部公告
- **以便** 各處室之間不互相干擾、且符合資料最小揭露原則

**驗收條件**
- **Given** 小美用教務處帳號登入
- **When** 她進入後台首頁
- **Then** 預設只看到教務處相關公告 (含她發的與其他教務處承辦發的);學務處的內部公告不會出現在她的列表 (除非她有跨處室閱讀權)

### 2.2 佐藤同學 (高二學生)

#### US-2.1 5 秒判斷「跟我有沒有關」
- **身為** 佐藤同學
- **我想要** 在公告列表中,5 秒內判斷「這則跟我有沒有關」
- **以便** 跳過「跟我無關」的社團招新、老師研習、總務採購公告

**驗收條件**
- **Given** 佐藤同學用手機登入 (Android 中階機)
- **When** 他進入公告列表
- **Then** 預設看到的都是「[高二]」或 [自然組] (他選的組) 相關公告,標題列同時顯示標籤 chip (例: [高二] [自然組] [考試]),非相關公告需手動展開「全部公告」才看得到

#### US-2.2 多標籤 AND 篩選:高二 + 自然組
- **身為** 佐藤同學
- **我想要** 用 [高二] AND [自然組] 兩個條件同時篩選公告
- **以便** 學期末看「下學期課表預排」時只看到跟我年級 + 選組相關的

**驗收條件**
- **Given** 佐藤同學在篩選器頁面
- **When** 他選擇 [高二] (年級標籤) + [自然組] (選組標籤) 兩個篩選條件
- **Then** 列表只顯示「同時具備 [高二] 與 [自然組] 兩個標籤」的公告;他可一鍵儲存此篩選為「我的常用篩選」下次直接套用

#### US-2.3 多標籤 OR 篩選:模擬考 OR 升學
- **身為** 佐藤同學
- **我想要** 用 [模擬考] OR [升學] 兩個條件篩選
- **以便** 一次看到所有「跟升學有關」的公告 (不論是模擬考或升學資訊)

**驗收條件**
- **Given** 佐藤同學在篩選器頁面
- **When** 他選擇 [模擬考] OR [升學] 兩個活動類型標籤
- **Then** 列表顯示「具備 [模擬考] 或 [升學] 任一標籤」的公告

#### US-2.4 巢狀篩選:高三模擬考 OR 高三升學 (參考)
- **身為** 佐藤同學
- **我想要** 進階篩選:([模擬考] OR [升學]) AND [高三]
- **以便** 雖然我高二但可以查「如果我升高三,會看到什麼」

**驗收條件** (此為巢狀篩選的展示性案例,實務上佐藤不會查高三公告;但功能要支援)
- **Given** 佐藤在進階篩選模式
- **When** 他設定 group 1 = [模擬考] OR [升學],group 2 = [高三],兩 group 之間用 AND 連接
- **Then** 系統正確顯示「同時具備 [高三] 且 (具備 [模擬考] 或 [升學])」的公告

#### US-2.5 附件下載後找得到
- **身為** 佐藤同學
- **我想要** 從公告下載的附件 (PDF 報名表) 在手機上「找得到」
- **以便** 不用從手機 Download 資料夾大海撈針

**驗收條件**
- **Given** 佐藤從公告點擊附件下載
- **When** 下載完成
- **Then** 系統提示「已存到我的附件」並列出此附件名稱、來源公告、下載時間;「我的附件」頁面可再次下載或開啟

### 2.3 陳媽媽 (家長會代表)

#### US-3.1 只看「家長」身分公告
- **身為** 陳媽媽
- **我想要** 預設只看到 [家長] 身分的公告
- **以便** 跳過學生自治會、教師研習、總務採購等跟我無關的訊息

**驗收條件**
- **Given** 陳媽媽用手機登入
- **When** 她進入公告列表
- **Then** 預設只看到標籤含 [家長] 的公告;若想看全部可切換到「全部公告」頁籤

#### US-3.2 附件 PDF 報名表下載 + 加提醒
- **身為** 陳媽媽
- **我想要** 看到 PDF 報名表 (校外教學、營隊) 能一鍵下載,並加入行事曆
- **以便** 不漏掉截止日、不用手動抄寫日期

**驗收條件**
- **Given** 陳媽媽看到一則含 PDF 報名表的公告 (例:高三校外教學)
- **When** 她點開附件
- **Then** 可線上預覽 PDF、可下載到「我的附件」、若公告有設定截止日,系統提示「加入行事曆」按鈕可一鍵加到 iOS / Android 行事曆

#### US-3.3 訂閱特定處室 (教務 + 學務)
- **身為** 陳媽媽
- **我想要** 訂閱 [教務處] + [學務處] 兩個處室的公告
- **以便** 不用看全部,只看我關心的處室

**驗收條件**
- **Given** 陳媽媽在「訂閱管理」頁面
- **When** 她勾選 [教務處] + [學務處]
- **Then** 之後這兩個處室發的公告會推播給她,其他處室 (總務、輔導) 不會推播;她仍可手動到「全部公告」看其他處室

#### US-3.4 簽收 (已簽) :  模擬考同意書
- **身為** 陳媽媽
- **我想要** 收到「需簽收」的公告 (例:模擬考同意書) 時,能明確點選「我已確認」
- **以便** 學校有我的簽收紀錄、不會再打電話來催

**驗收條件**
- **Given** 陳媽媽收到一則標籤為 [需簽收] 的公告
- **When** 她點開公告、捲到最下方點選 [我已確認簽收] 藍色按鈕
- **Then** 系統記錄簽收時間,學校後台顯示「此家長已簽」,陳媽媽的「我的簽收紀錄」也新增一筆

#### US-3.5 截止日提醒
- **身為** 陳媽媽
- **我想要** 對有截止日的公告 (例:繳費只到週五) 收到主動提醒
- **以便** 不會過期才被通知

**驗收條件**
- **Given** 陳媽媽已讀一則有截止日的公告 (截止日為 3 天後)
- **When** 系統進入「截止日前 24 小時」時間點
- **Then** 推播通知「[高三] 校外教學報名將於明天截止,請儘早完成繳費」

## 3. 資訊架構 (IA)

### 3.1 頁面樹 (4 個頂層區)

```
校園公告系統
├── / (首頁 :  登入後導向個人化公告列表)
├── /announcements (公告區)
│   ├── /announcements/list (公告列表 :  預設篩選「跟我有關」)
│   ├── /announcements/:id (公告詳情)
│   ├── /announcements/search (搜尋結果)
│   └── /announcements/new (新增公告 :  限處室承辦以上)
├── /tags (標籤管理 :  限系統管理員)
│   ├── /tags/list (所有標籤列表)
│   ├── /tags/:id (單一標籤詳情 :  顯示名稱、類型、用法、引用此標籤的公告數)
│   └── /tags/new (新增標籤)
├── /admin (後台)
│   ├── /admin/dashboard (儀表板 :  我的公告、簽收統計、未讀名單)
│   ├── /admin/announcements (我的公告 CRUD)
│   ├── /admin/attachments (附件管理)
│   ├── /admin/users (使用者管理 :  限系統管理員)
│   ├── /admin/roles (角色權限管理 :  限系統管理員)
│   └── /admin/audit (稽核紀錄 :  限系統管理員)
└── /me (個人區)
    ├── /me/profile (個人資料)
    ├── /me/subscriptions (訂閱管理)
    ├── /me/attachments (我的附件)
    ├── /me/signatures (我的簽收紀錄)
    └── /me/filters (我的常用篩選)
```

### 3.2 主要導航

**頂部列 (全部頁面共用)**
- 左:系統 Logo + 學校名稱
- 中:全域搜尋框 (標題 / 內文 / 附件檔名)
- 右:訂閱管理、通知中心 (鈴鐺 icon + 未讀數)、個人頭像 (下拉:個人資料 / 切換身分 / 登出)

**側邊欄 (依角色顯示)**
- 學生 / 家長 / 教師:[公告列表] [我的附件] [我的簽收] [我的常用篩選] [訂閱管理]
- 處室承辦 (小美類):多了 [我的公告後台] [新增公告] [簽收統計]
- 系統管理員:再多了 [標籤管理] [使用者管理] [角色權限] [稽核紀錄] [處室設定]

### 3.3 主要路由表

### 3.4 行動裝置導航差異

- 桌機:左側固定側邊欄
- 手機:側邊欄收合為漢堡選單 (三條線),頂部列保留 Logo + 搜尋 + 通知 + 頭像
- 公告列表在手機上為「單欄卡片」,每則公告顯示 1-2 個最重要的標籤 chip (其餘以 +N 表示)

## 4. 核心功能詳細化

### 4.1 公告 CRUD

#### 4.1.1 資料模型 (邏輯欄位)

#### 4.1.2 列表頁功能

- 排序:依發布時間倒序 (新到舊)
- 分頁:每頁 20 則,桌機可選 50/100
- 篩選入口:點「進階篩選」展開標籤選擇器
- 顯示欄位:標題、發布者、發布時間、標籤 chip (最多 3 個)、附件 icon、需簽收 badge、未讀 / 已讀狀態

#### 4.1.3 新增 / 編輯頁功能

- 標題輸入框 (1-200 字,即時字數)
- 內文輸入框 (純文字,支援換行)
- 標籤選擇器 (見 §4.2.4)
- 附件上傳區 (見 §4.4)
- 「需簽收」勾選框 + 簽收截止日 (勾選才顯示)
- 受眾設定 (MVP 簡化為標籤即受眾,不必另設「指定受眾」欄位)
- 預覽按鈕 (在儲存前看公告長相)
- 送出按鈕

#### 4.1.4 刪除

- 軟刪除 (status 改為 archived 或 deleted_at 填值)
- 刪除前二次確認對話框
- 刪除後從所有使用者列表消失,但系統管理員可在稽核紀錄看到

#### 4.1.5 查詢 / 搜尋

- MVP 全文搜尋:標題、內文、附件檔名 (case-insensitive,部分相符)
- v2 之後才做進階搜尋 (依日期範圍、發布者、標籤組合)

### 4.2 標籤系統 (核心,佔大篇幅)

> **為什麼是核心**:consumer-researcher 把「標籤 OR/AND 篩選」列為 Must 的理由是「這是台灣校園公告比 Notion 2020-05 之前更落後的地方」,也是佐藤同學 (5 秒判斷跟我有沒有關) 與陳媽媽 (訂閱特定處室) 的核心痛點。沒有標籤,後續的「已讀 / 已簽」、「訂閱」、「篩選」全部失效。

#### 4.2.1 標籤類型 (5 種,不可混用)

| 類型 | 說明 | 範例 | 誰可建立 |
|------|------|------|----------|
| grade | 年級 | [高一] [高二] [高三] | 系統管理員 |
| class | 班級 | [一年一班] [二年三班] | 系統管理員 |
| department | 處室 | [教務處] [學務處] [總務處] [輔導處] [校長室] | 系統管理員 |
| activity | 活動類型 | [模擬考] [升學] [研習] [競賽] [課表異動] [收費] [營隊] [防疫] | 系統管理員 + 處室承辦 (可建議新增,系統管理員核准) |
| role | 受眾身分 | [學生] [家長] [教師] [訪客] | 系統管理員 |

> 注意:本 MVP 標籤類型採「扁平分類」,不支援巢狀 (例:不能有 [高一] 之下再分 [一年一班])。若需巢狀,屬於 v2 (見 §6.1)。

#### 4.2.2 標籤資料模型 (邏輯欄位)

| 欄位 | 型別 | 必填 | 說明 |
|------|------|------|------|
| id | UUID | 自動 | |
| type | enum | 是 | 5 種類型之一 (見 §4.2.1) |
| name | 字串 | 是 | 標籤顯示名稱,同一 type 內不可重複 |
| description | 字串 | 否 | 標籤說明 (例:[模擬考] = 全校統一模擬考相關) |
| color | 字串 | 否 | UI 顯示色 (hex),系統管理員可設定 |
| created_by | UUID | 自動 | 建立者 |
| created_at | datetime | 自動 | |
| is_active | bool | 自動 | 是否啟用 (停用後不可掛新公告,既有公告不受影響) |

#### 4.2.3 公告與標籤的多對多關聯

- 一則公告可掛 0 個到 N 個標籤
- 一個標籤可掛在 0 個到 N 個公告
- 關聯表邏輯欄位:announcement_id、tag_id、created_at
- 一則公告最少要有 1 個標籤才能發布 (系統強制)
- 一則公告最多 10 個標籤 (避免過度標籤化)
- 同類型標籤 (例:[教務處] [學務處] 都是 department) 不限制數量;但實務建議 2-3 個為佳
- 不同類型標籤建議至少各 1 個:例 [高三] (grade) + [模擬考] (activity) + [學生] (role) = 1 公告,精準對應 3 種維度的受眾

#### 4.2.4 標籤選擇器 (UI 行為)

- 在「新增公告」頁面,標籤區分為 5 個區塊 (對應 5 種類型)
- 每個區塊顯示該類型下所有啟用中的標籤,以 chip 形式呈現
- 點 chip = 選取,再點 = 取消選取
- 選取中的 chip 高亮、顯示已選數量 / 該類型總數
- 提供「搜尋標籤」輸入框,可過濾標籤名稱 (例:輸入「模擬」即顯示 [模擬考])
- 跨類型可同時選 (例:同時選 [高三] (grade) + [模擬考] (activity) + [學生] (role))
- 已選標籤總數即時顯示在標籤區塊頂部

#### 4.2.5 篩選邏輯 (核心: OR / AND / Nested / NOT)

##### 4.2.5.1 基本 OR 篩選 (同類型內多選為 OR)

- **情境**:勾選 [模擬考] + [升學] (都是 activity 類型)
- **語意**:`公告.標籤 ⊇ {[模擬考] 或 [升學]}`(任一)
- **對應 SQL 邏輯概念**:`EXISTS(tag WHERE name='模擬考') OR EXISTS(tag WHERE name='升學')`
- **驗收**:列表顯示「具備 [模擬考] 或 [升學] 任一標籤」的公告

##### 4.2.5.2 基本 AND 篩選 (跨類型間為 AND)

- **情境**:勾選 [高二] (grade) + [自然組] (role,本 MVP 簡化為 role)
- **語意**:`公告.標籤 ⊇ {[高二] AND [自然組]}`(兩者皆備)
- **對應 SQL 邏輯概念**:`EXISTS(tag WHERE name='高二') AND EXISTS(tag WHERE name='自然組')`
- **驗收**:列表顯示「同時具備 [高二] 與 [自然組] 兩個標籤」的公告

##### 4.2.5.3 巢狀 (Nested) 篩選 (進階)

- **情境**:([模擬考] OR [升學]) AND [高三]
- **語意**:同類型 (activity) 內用 OR、跨類型 (activity 與 grade) 用 AND
- **UI 行為**:提供「進階篩選」模式,可新增多個「篩選 group」,每個 group 內的條件可選 AND 或 OR,group 之間再用 AND 或 OR 連接
- **對應 SQL 邏輯概念**:類似 Notion 2020-05 之後的 Nested Filters (`(filter1 AND filter2) OR (filter3 AND filter4)`)
- **驗收**:系統正確顯示「同時具備 [高三] 且 (具備 [模擬考] 或 [升學])」的公告
- **MVP 簡化**:MVP 僅支援「group 內 OR,group 間 AND」(最常見的情境);完整自由巢狀 (group 內 AND + group 間 OR) 屬於 v2

##### 4.2.5.4 NOT 排除篩選

- **情境**:[高三] AND NOT [模擬考]
- **語意**:只看高三、但排除模擬考
- **UI 行為**:在標籤 chip 上有「排除」選項 (例:點 chip 的 × = 加入排除條件)
- **驗收**:列表顯示「具備 [高三] 且 不具備 [模擬考]」的公告
- **MVP 簡化**:MVP 僅支援「同類型內 NOT」,跨類型 NOT 屬於 v2

#### 4.2.6 篩選器 UI 雙軌設計 (對標 Gmail 對照組 A)

- **入門模式**:勾選式 UI,系統自動依「同類型 OR、跨類型 AND」組合
- **進階模式**:可手動編輯巢狀 group、NOT 條件
- **儲存常用篩選**:使用者可把目前篩選條件存為「我的常用篩選」(例:「我的課表」、「我的活動」),之後一鍵套用

#### 4.2.7 標籤管理後台 (限系統管理員)

- 路徑:/tags/list 與 /tags/:id
- 功能:
  - 新增標籤 (選類型 + 填名稱 + 填說明 + 選顏色)
  - 編輯標籤 (改說明、改顏色)
  - 停用標籤 (is_active=false,既有公告不受影響,新公告不可掛)
  - 啟用標籤
  - 刪除標籤 (需二次確認,且只能刪「無公告引用」的標籤;若有公告引用,改為「停用」)
  - 查看「此標籤被 N 則公告引用」(點進去看公告列表)
- 稽核:任何標籤的建立 / 編輯 / 停用 / 刪除都進入稽核紀錄

#### 4.2.8 標籤與其他功能的關聯

- **公告 CRUD**:見 §4.1,每則公告必掛至少 1 個標籤才能發
- **已讀 / 已簽追蹤**:小美在後台可「依標籤」篩選簽收狀態 (例:只看 [高三] 的簽收率)
- **推播通知**:見 §4.2.9
- **訂閱管理**:見 §4.2.10
- **搜尋**:全文搜尋結果可再用標籤二次篩選

#### 4.2.9 標籤與推播的關係 (M-08 推播的具體化)

- 預設推播規則:公告發布時,所有「具備公告任一標籤」的使用者收到推播
- 例:公告掛 [高三] (grade) + [模擬考] (activity) + [學生] (role) = 所有具備這 3 個標籤中任一的使用者都收到
- 例外:使用者可在「訂閱管理」設定「只收 [教務處] 或 [學務處]」,其他標籤不推播

#### 4.2.10 標籤與訂閱的關係 (S-04 訂閱的具體化)

- 雖然「訂閱」是 v2 Should,但標籤系統必須為此預留設計空間
- 訂閱的本質是「過濾條件」,與篩選器結構相同 (見 §4.2.5)
- MVP 階段:使用者可在「個人化首頁」設 1 個預設篩選 (= 訂閱的簡化版)
- v2 階段:擴展為多組訂閱,每組可設名稱、推播開關

### 4.3 各處室獨立登入

#### 4.3.1 設計原則

- 每個處室是一個獨立的「namespace」(邏輯隔離)
- 同處室承辦看到的公告 = 該處室發的 + 跨處室公開公告
- 不同處室之間公告預設不互通 (符合「資料最小揭露」與「處室獨立運作」)
- 系統管理員可設「跨處室閱讀權」(例:教務主任可讀學務處公告)

#### 4.3.2 預設建立的處室帳號 (系統建置時一次性 seed)

| 處室 | 預設帳號 | 預設密碼 | 預設角色 | 預設可發布的標籤類型 |
|------|----------|----------|----------|---------------------|
| 教務處 | academic.officer@school | 系統初始化時產生一次性密碼 | 處室承辦 | grade (年級) + class (班級) + activity (教務類) |
| 學務處 | student.affairs@school | 同上 | 處室承辦 | activity (學務類:競賽、活動) |
| 總務處 | general.affairs@school | 同上 | 處室承辦 | activity (總務類:採購、修繕) |
| 輔導處 | counseling@school | 同上 | 處室承辦 | activity (輔導類:升學、輔導) |
| 校長室 | principal@school | 同上 | 系統管理員 | 全部 |
| 資訊組 (系統管理) | sysadmin@school | 同上 | 系統管理員 | 全部 |

- 一次性密碼:首次登入強制變更
- 帳號命名:`<英文簡稱>.officer@school`(處室承辦) 或 `<英文簡稱>@school`(其他)
- 每個處室可有多個承辦帳號,系統管理員可新增

#### 4.3.3 登入流程

- 單一登入入口 (例:https://bulletin.school.edu.tw/login)
- 輸入 email + 密碼
- 驗證成功後,token 帶入「所屬處室」資訊,前端依此顯示對應的側邊欄與後台
- 登入失敗:顯示「帳號或密碼錯誤」(不分別提示,避免帳號枚舉)
- 忘記密碼:email 重設連結 (連結 1 小時內有效)
- Session 時效:MVP 設 8 小時 (對應上半天上班);v2 改為滑動式 8 小時

#### 4.3.4 角色權限矩陣 (M-06 的具體化)

| 角色 | 看公告 | 發公告 | 編自己公告 | 編別人公告 | 刪自己公告 | 刪別人公告 | 管理標籤 | 管理使用者 | 看稽核 |
|------|--------|--------|------------|------------|------------|------------|----------|------------|--------|
| 系統管理員 | 全部 | 可 | 可 | 可 | 可 | 可 | 可 | 可 | 可 |
| 處室承辦 | 該處室 + 公開 | 可 (限本處室) | 可 | 不可 | 可 | 不可 | 不可 | 不可 | 不可 |
| 教師 | 全部 (依標籤過濾) | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 |
| 家長 | 全部 (依標籤過濾) | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 |
| 學生 | 全部 (依標籤過濾) | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 |
| 訪客 | 只看 public 公告 | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 | 不可 |

- 「依標籤過濾」:使用者只看到「具備自己任一身分 / 年級 / 班級 / 訂閱處室標籤」的公告
- 訪客 (未登入) MVP 不開放 (本 MVP 假設全學校人員都有帳號)

#### 4.3.5 跨處室公告

- 一則公告可掛多個處室標籤 (例:[教務處] + [學務處] 聯合發)
- 此時公告歸屬於「發布者的處室」,但受眾包含兩個處室訂閱者
- 編輯權限仍限原發布者 + 系統管理員

### 4.4 附件上傳與下載

#### 4.4.1 支援的檔案格式 (對標 Cloud School 12+ 格式)

- 文件:`pdf`, `doc`, `docx`, `odt`
- 試算表:`xls`, `xlsx`, `ods`, `csv`
- 簡報:`ppt`, `pptx`, `odp`
- 圖片:`jpg`, `jpeg`, `png`, `gif`, `webp`
- 壓縮檔:`zip`, `7z` (上限內)
- 其他:`txt`, `md`

> 不支援可執行檔 (`exe`, `bat`, `sh`) 與巨集文件 (`xlsm`, `docm`) :  資安考量

#### 4.4.2 大小限制

- 單檔上限:50 MB (對標 email 25MB 的 2 倍,留 buffer)
- 單則公告附件總和上限:200 MB
- 超出限制:前端即時提示,不得上傳

#### 4.4.3 上傳流程 (對標 Cloud School 兩階段)

- **階段 1 :  上傳到儲存層**:前端選擇檔案後,系統先將檔案上傳至儲存層 (物件儲存),取得「附件 ID」與「臨時 URL」
- **階段 2 :  綁定到公告**:在「新增公告」頁面,使用者可從「已上傳的附件」中選擇要綁定到本則公告的附件 (複選)
- 兩階段的好處:同一附件可重複用於多則公告,不必重傳;附件可預先上傳、事後再綁定
- 上傳中:顯示進度條 (百分比 + 已上傳 / 總大小)
- 上傳失敗:顯示錯誤訊息 (例:網路中斷、檔案太大、格式不支援),允許重試

#### 4.4.4 附件列表顯示

- 公告詳情頁的「附件」區,列出所有附件:
  - 檔名 (可點擊下載)
  - 副檔名 icon (PDF / Word / Excel / 圖片 / 其他)
  - 檔案大小 (KB / MB)
  - 上傳時間
  - 上傳者 (小美類)
- 點檔名 = 下載;點 icon = 線上預覽 (v2 才有, MVP 下載為主)

#### 4.4.5 下載行為

- 一般下載:點檔名瀏覽器直接下載
- 圖片附件:可選擇「下載」或「線上預覽」(MVP 提供此基本預覽)
- PDF 附件:可選擇「下載」或「線上預覽」(MVP 提供此基本預覽,使用瀏覽器原生 PDF 檢視器)
- Word / Excel 附件:僅下載,不預覽 (v2 才做 Office online 整合)

#### 4.4.6 附件權限

- 附件跟隨公告的標籤受眾:看不到公告的人也看不到附件
- 「我的附件」頁面:使用者看自己曾下載過的所有附件 (含原始公告連結),不過期

#### 4.4.7 附件生命週期

- 公告存在期間:附件可下載
- 公告軟刪除 (archived):附件仍保留 30 天供稽核,之後由系統管理員手動刪除
- 公告永久刪除 (系統管理員操作):附件一併刪除

#### 4.4.8 附件版本 (v2)

- MVP 不支援附件版本歷史 (屬於 S-06 Should)
- 介面預留:附件列表可顯示「v1」「v2」(目前永遠顯示 v1)

## 5. 非功能需求

### 5.1 效能需求

| 指標 | 目標值 | 備註 |
|------|--------|------|
| 公告列表首頁載入 | < 1.5 秒 (4G) | 桌機 < 0.8 秒 |
| 公告詳情頁載入 | < 1.0 秒 | |
| 標籤篩選套用 | < 0.5 秒 | 即時 client-side 套用,不必送後端 |
| 附件下載啟動時間 | < 2.0 秒 (50MB 檔) | 端對端下載速度取決於網路 |
| 搜尋結果 | < 1.5 秒 | |
| 推播送達時間 | < 30 秒 (從發布到使用者手機) | |
| 公告並發上線承載 | 100 位使用者同時上線 | MVP 對應單校 5,000 學生 + 教職員 |

### 5.2 資安需求

- **傳輸加密**:全站 HTTPS, TLS 1.2 以上
- **密碼儲存**:bcrypt 或同等強度雜湊 (不儲明文), salt 至少 16 byte
- **Session 管理**:HttpOnly + Secure cookie,過期自動登出
- **CSRF 防護**:所有寫入操作 (POST / PATCH / DELETE) 帶 CSRF token
- **XSS 防護**:使用者輸入的公告標題、內文、附件檔名在渲染前 escape;不接受 raw HTML
- **SQL Injection**:所有查詢使用參數化查詢
- **附件掃毒**:上傳後非同步掃毒 (MVP 至少串接一個 AV engine, ClamAV 或雲端掃毒 API)
- **附件存取控制**:附件 URL 帶時效 token,過期失效 (見 §5.2 附件下載連結)
- **稽核紀錄**:系統管理員的任何操作 (建立 / 編輯 / 停用 / 刪除標籤、使用者管理、角色變更) 全部寫入稽核 log,保留 1 年
- **個資保護**:附件中若有個資 (PDF 報名表含姓名 / 電話),下載行為寫入稽核 (誰、何時、下載了什麼)
- **個資法遵循**:使用者可要求匯出 / 刪除自己的資料 (被刪除使用者的歷史已讀 / 已簽紀錄匿名化保留)

### 5.3 可用性需求

- **WCAG 2.1 AA 等級**:鍵盤可操作、螢幕閱讀器可讀、顏色對比 4.5:1
- **語言**:繁體中文 (zh-TW) 為主,英文為輔 (後台介面)
- **瀏覽器支援**:Chrome / Edge / Safari / Firefox 最新兩個版本
- **行動作業系統**:iOS 15+ / Android 10+
- **RWD 斷點**:桌機 (>1024px) / 平板 (768-1024px) / 手機 (<768px) 三套佈局
- **錯誤訊息**:使用者友善,不暴露技術細節;提供建議動作
- **空狀態**:列表無資料時顯示「目前沒有公告」+ 圖示,不是空白頁
- **Loading 狀態**:列表載入中顯示 skeleton (灰底佔位)
- **離線**:MVP 不支援離線瀏覽 (v2 才做 PWA)
- **操作中斷防護**:上傳附件中斷網路,提示「上傳失敗,請重試」並保留已選檔案名

### 5.4 部署環境 (建議範圍,非技術選型)

- **目標環境**:單一學校 (中型高中, 學生 1,000-3,000 人、教職員 100-300 人)
- **可用性目標**:99.5% (允許每月 4 小時 downtime,對應上班時間外的維護窗口)
- **資料備份**:每日全備份 + 每小時增量,備份保留 30 天
- **災難恢復**:RPO 1 小時 (最多遺失 1 小時資料),RTO 4 小時 (4 小時內可恢復服務)
- **上線模式**:新校導入時,可選擇「雲端託管」或「校內機房自建」兩種模式
- **瀏覽器建議**:不需要裝 APP,純 RWD 網頁即可 (降低導入門檻;v2 才考慮 PWA / 原生 APP)
- **多租戶**:MVP 不支援多校共用 (一所學校一個 instance)
- **時區**:Asia/Taipei (UTC+8)
- **語系**:zh-TW 為主
- **本機時間格式**:YYYY/MM/DD HH:mm
- **檔案編碼**:UTF-8

### 5.5 可維護性

- **文件**:本 PRD + 給 system-architect 的接力清單 (§7) + 上線後的 runbook (由 system-architect 接手後另寫)
- **錯誤日誌**:前端 console error + 後端 application log,集中收集
- **監控指標**:頁面 PV / UV、API 響應時間、推播送達率、附件上傳成功率、登入失敗率
- **版本升級**:採語意化版本號 (semver),升級前提供 release notes

## 7. 給下一棒 (system-architect) 的接力清單

> 本節只「點出 system-architect 該考慮的問題」,**不給答案**(答案屬於技術選型)。

### 7.1 標籤 OR/AND 的 query 設計

**該考慮的問題**:
- 公告與標籤是「多對多」關係,join 查詢的效能 (尤其當公告數破萬、標籤數破百時)
- 同類型 OR、跨類型 AND、巢狀 (group 內 OR、group 間 AND)、NOT 排除 :  這四種組合如何用一套查詢結構表達
- 索引設計:公告 ID + 標籤 ID 複合索引、標籤 type + name 索引
- 篩選是「client-side 即時套用」還是「送後端 query」?若 client-side,資料要先全部載入;若送後端,需考量 query 快取
- 全文搜尋 + 標籤篩選的交集:是先全文搜再標籤過濾,還是反向?
- 巢狀篩選的 UI 操作複雜度 vs 後端 query 複雜度取捨

### 7.2 附件儲存方案

**該考慮的問題**:
- 自建儲存 (本地檔案系統) vs 雲端物件儲存 (S3 相容)
- 大檔 (50MB) 上傳:分段上傳 (multipart upload) 是否需要
- 附件與公告的關聯:上傳後就永久綁定?還是可重新綁定到其他公告?
- 附件的「過期 / 軟刪除 / 永久刪除」生命週期如何實作
- 附件下載的時效 token 怎麼生 (v2 議題,但 MVP 也要預留)
- 附件掃毒 (ClamAV / 雲端掃毒) 的整合方式:同步掃 vs 異步掃?掃毒失敗時附件如何處理
- 附件的存取控制:URL 直連 vs 經後端 proxy?經後端可擋未授權存取但增加後端負載
- 附件版本 (v2 議題) 的儲存結構:同檔多版本是覆蓋還是另存

### 7.3 認證 / 授權 (Auth) 方案

**該考慮的問題**:
- 「各處室獨立登入」≠「各處室完全獨立系統」,而是「同一系統、不同 namespace」
- SSO (單一登入) 是否要做?若日後要跟校務系統、Google Workspace、Microsoft 365 整合?
- 密碼策略:最少 8 字元 + 英文數字混用?是否要 2FA (OTP / 簡訊)?
- 角色權限分離 (5+ 層) 用 RBAC 模型還是 ABAC?是否需要可程式化擴充 (對標 WordPress `add_cap`)
- 跨處室閱讀權的設定:是 hard-code 還是後台可設定?由誰設定 (系統管理員 / 處室主任)?
- 家長 / 學生帳號的生命週期:每學年重置?離校 / 畢業後如何停用?
- Token 管理:JWT vs session cookie?Refresh token 機制?
- 登入失敗的 rate limiting:連續失敗 N 次鎖定?
- 訪客 (未登入) 訪問 public 公告的體驗

### 7.4 推播通知 (M-08) 的設計

**該考慮的問題**:
- Web Push (瀏覽器原生推播) vs FCM (Android) / APNs (iOS) vs Line Notify / Telegram Bot
- 推播送達率 vs 開發成本取捨
- 「訂閱」機制 (v2-04) 如何與推播整合
- 推播的節流:同一人 5 分鐘內最多收幾則?避免洗版
- 推播失敗的重試機制

### 7.5 已讀 / 已簽雙軌制的實作

**該考慮的問題**:
- 「已讀」與「已簽」是兩個獨立事件,如何記錄 (同一 table 不同欄位?還是兩個 event?)
- 公告詳情頁開啟 = 自動觸發已讀?還是要使用者點「已讀」按鈕?
- 已簽的「法律效力」問題:v3 電子簽章之前,系統簽收記錄算「私證書」還是「準公證書」?
- 未讀者的「主動提醒」:提醒頻率上限?Line 通知 vs email vs 站內信?

### 7.6 行動裝置原生體驗 (M-09) 的設計

**該考慮的問題**:
- RWD 網頁 vs PWA vs 原生 APP (iOS / Android) 三條路如何選
- 推播穩定性:RWD 網頁的推播需要 service worker,使用者首次訪問要允許通知
- 離線瀏覽 (v3) 的資料快取策略
- 安裝提示:如何讓使用者把網頁加到主畫面 (PWA)

### 7.7 多租戶 / 部署架構

**該考慮的問題**:
- MVP 假設單一學校一個 instance,若日後要支援多校共用 (multi-tenant) 怎麼擴展
- 雲端託管 vs 校內機房自建的成本與維運差異
- 資料隔離:不同學校的資料是否共用同一個 database 但用 tenant_id 區隔,還是各自獨立 database

### 7.8 資料模型與資料庫

**該考慮的問題**:
- 公告、標籤、使用者、處室、附件的實體關係圖
- 標籤的 5 種類型 (grade / class / department / activity / role) 是用單一 table + type 欄位,還是 5 個 table
- 軟刪除 (soft delete) 的實作:deleted_at 欄位 vs 狀態機
- 稽核紀錄的儲存:同 database 不同 table?還是獨立的 audit log service
- 全文搜尋:用 RDBMS 內建全文搜尋 還是獨立的搜尋引擎?兩者在索引維護成本、查詢效能、多語系支援的取捨為何?

### 7.9 稽核與法遵

**該考慮的問題**:
- 個資法的「資料可攜」與「被遺忘權」如何實作
- 附件中若有個資 (PDF 報名表含身分證字號),下載行為的稽核粒度
- 兒少法 (本系統使用者包含未成年學生) 的特殊要求
- 保留政策:公告、附件、稽核 log 的保存年限
