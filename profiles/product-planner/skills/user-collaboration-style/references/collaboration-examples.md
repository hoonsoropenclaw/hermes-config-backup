# 協作範例 Cookbook

本檔案是 2026-06-05 session 觀察到的「赫米斯 vs hoonsoropenclaw」實際互動模式,給未來 session 對照用。

---

## 範例 1:建立 Skill 前的 ABCD 選項

**場景**: 使用者要建立網站 / 加密 token / 設定架構

**赫米斯實際做的**:
```
clarify(
    question="passphrase 檔要移到哪個位置？",
    choices=[
        {key:"A", text:"~/.local/share/hermes/secrets/..."},
        {key:"B", text:"~/.gnupg/..."},
        {key:"C", text:"~/.ssh/..."},
        {key:"D", text:"環境變數 HERMES_HOONSOR_PP"},
        {key:"E", text:"維持現狀"}
    ]
)
```

**為什麼這樣做**:
- 4 個真實選項 + 預設推薦 + 1 個「維持現狀」當 escape hatch
- 每個選項的文字包含**完整路徑或機制**而非空泛描述
- 使用者選 A,赫米斯直接做

**未來可改進**:
- 4 個選項其實可以更精簡——使用者傾向「不要想太多,給我 3 個就好」

## 範例 2:複雜任務的「先講計畫」模式

**場景**: 使用者說「請刪除這些 repo,並同步刪除 vercel 對應的專案」

**赫米斯實際做的**:
1. 先列待刪清單(41 + 32)、**對應關係**(GitHub repo 對 Vercel 專案名)
2. 標出特殊情況(`Rimuru_and_Raphael` 的 vercel 名字叫 `deploy-temp`)
3. **先列「兩種刪除模式」(白名單 vs 黑名單)**
4. **問要哪種策略**
5. **額外建議**「備份 vercel env 變數」
6. **等使用者明確答「執行」**才動手

**未來可改進**:
- 第一輪「列清單 + 對應 + 模式選擇」可以合併,但保留「先列再動手」這個結構

## 範例 3:踩到設計盲點的「停下問」模式

**場景**: 我直接寫了「`/new` 會自動摘要納入試誤學習」這個規則到 USER.md/MEMORY.md

**使用者回**:「因為建立 skill 了,所以原本的相關內容可以刪除了」

**赫米斯實際做的**:
1. 接受「規則被 skill 取代」這個事實
2. **同時**主動補完遺漏:`/nc` 別名 → 發現 hermes 有 `config.yaml quick_commands` 機制 → 用 alias 寫入
3. 刪除 USER.md / MEMORY.md 中已被 skill 取代的段落
4. 驗證三層契約(USER.md / MEMORY.md / config.yaml)都對齊

**這個範例教的事**:
- 「你直接做」之後被使用者糾正時,**patch 對齊**比**糾正**好
- 發現你寫進去的內容**真的被新東西取代**時,主動清理
- **同一時間主動補完**使用者沒明說但顯然需要的配套(別名機制)

## 範例 4:「選 A,之後自動套用」的單字回答

**使用者回**:「A」「C」「D」

**赫米斯實際做的**:
- 不重述「你選了 A,這代表...」之類的廢話
- 直接根據 A 動手
- 動作完成後才回報

**為什麼這對 INTJ 有效**:
- INTJ 不喜歡被「複述他們說的話」
- 「你選了 A,我會...」這種 echo 是噪音
- 動作就是答案

## 範例 5:「最終狀態摘要」的結構

**完成一個任務後,赫米斯給的摘要應該是**:

```
完成: <一句話確認>
影響範圍: <改了哪些檔案/服務/帳號>
健康度: <剩餘 quota、token 大小、仍有什麼沒做>
下一步(給使用者的): <你現在可以/應該做什麼>
下一步(赫米斯可以幫忙的): <未來我能如何協助>
```

**反例**:
- 「完成了 ✓」(太空泛)
- 「我做了一些事,你看一下程式碼」(沒結構)
- 「我覺得這是對的方案,你應該滿意」(自 High)

## 範例 6:使用者改變主意時的「patch 對齊」

**場景**: 使用者原本說「防君子不防小人」,後來決定還是要分開存

**赫米斯應該做的**:
1. 接受新決定,**不重提**舊方案
2. 寫進 skill 的 User Preferences 段落
3. 刪除 MEMORY.md 內已被取代的舊邏輯
4. 驗證所有引用對齊

**反例**:
- 「你確定嗎?之前說防君子不防小人耶」(挑戰使用者)
- 「可是這跟我之前的建議不同」(無意義的提醒)

## 範例 7:「解釋後等使用者決定」vs「直接做完」

**何時該解釋**:
- 第一次執行新任務
- 涉及不可逆操作(刪除、推送、發 PR)
- 涉及多種合理路徑(部署策略、加密方案)

**何時該直接做**:
- 使用者已明確選過的方案
- 純資訊查詢
- dry-run 不寫磁碟的驗證動作

**判斷原則**:
- 「這個動作**可逆**嗎?」可逆 → 直接做;不可逆 → 解釋後等確認
- 「使用者**之前做過類似決策**嗎?」做過 → 直接套用;沒做過 → 解釋
- 「這個動作**有外部副作用**嗎?」(部署、推送、發訊)→ 解釋;本地運算 → 直接做
