# 技能搭配建議之技能 - 設計方向書

## 1. 願景

建立一個能「理解技能本質」並「根據任務情境推薦最佳技能組合」的技能系統。

不是簡單的「標籤匹配」，而是：
- **理解技能的能力邊界**：每個技能擅長什麼、不擅長什麼
- **理解任務的複雜度階梯**：簡單任務用輕量組合，複雜任務用多層協作
- **根據反饋持續學習**：記錄哪些組合有效、哪些無效，建立「技能組合知識庫」

---

## 2. 核心問題

現有的技能系統是「靜態分類」：
- 使用者看到 100+ 技能，不知道從哪裡下手
- 每次遇到新任務，只能靠直覺或 trial & error
- 沒有「組合視角」——技能 A + 技能 B 一起用會發生什麼？

赫米斯需要一個「動態技能組合推薦引擎」。

### 3.1 Skill Scanner（技能掃描器）
定期讀取 `~/.hermes/skills/` 目錄，建立技能元資料庫：
- 技能名稱、描述、category
- 觸發關鍵字（從 description 萃取）
- 依賴工具（從 SKILL.md 萃取）
- 預估執行時間、適用場景
- 歷史使用次數（從 cron job log 統計）

### 3.2 Composer Engine（組合引擎）
根據任務分析，推薦技能組合：

**推薦策略：**
1. **主技能**：承擔主要工作的技能
2. **輔助技能**：處理周邊任務（搜尋、檔案、日誌）
3. **驗收技能**：確保輸出品質（QA、測試）
4. **通訊技能**：向用戶回報結果

**複雜度分級：**
| 等級 | 描述 | 典型組合 |
|------|------|---------|
| L1 簡單 | 單一技能完成 | `web_search` → 回覆 |
| L2 一般 | 2-3 技能串聯 | `web_search` → `docx` → 回覆 |
| L3 複雜 | 主技能 + 輔助 + 驗收 | `scrapling` → `xlsx` → `site-qa-checklist` |
| L4 超複雜 | 多智能體並發協作 | `ai-agent-development` + `agent-orchestrator` + `tradingagents` |

### 3.3 Combo Registry（組合註冊表）
記錄所有嘗試過的技能組合：
```json
{
  "id": "combo-001",
  "skills": ["scrapling", "xlsx", "site-qa-checklist"],
  "task_type": "網頁資料擷取並生成報告",
  "created_at": "2026-06-01",
  "evaluation": {
    "score": 4.5,
    "reviewer": "user",
    "feedback": "xlsx 生成速度太慢，應改用 minimax-xlsx",
    "reviewed_at": "2026-06-02"
  }
}
```

### 3.4 Feedback Tracker（反饋追蹤器）
收集各管道的評價：
- **用戶直接反饋**：「效果好」/「不理想」
- **網路評價**：從 GitHub stars、相關討論萃取
- **自動化指標**：執行時間、錯誤率、Token 消耗

### 3.5 Combo Score Board（組合評分榜）
對所有註冊組合進行評分排名：
```json
{
  "combo_id": "scrapling-xlsx-site-qa",
  "avg_score": 4.2,
  "total_uses": 15,
  "last_used": "2026-06-10",
  "tags": ["資料擷取", "報表生成", "高評價"]
}
```

## 4. 使用者互動方式

### 4.1 主動推薦模式
當用戶描述一個任務時，主動推薦：
```
根據你的任務「分析競爭者網站並生成報告」，
我推薦以下技能組合：

🔧 主技能：scrapling（網頁資料擷取）
📊 輔助技能：minimax-xlsx（生成 Excel 報告）
✅ 驗收技能：site-qa-checklist（部署前 QA）
📝 通訊技能：docx（生成 Word 說明文件）

歷史評價：此組合已使用 12 次，平均 4.3 分
💡 小提醒：xlsx 生成速度較慢，如需快速產出可改用 docx
```

### 4.2 查詢模式
用戶可主動查詢：
- 「做 XXX 推薦什麼技能？」
- 「哪個組合評價最好？」
- 「有沒有做過 YYY 的案例？」

### 4.3 評價模式
任務完成後引導用戶評價：
```
任務完成！請幫這個技能組合評分：

組合：scrapling → minimax-xlsx → site-qa-checklist
任務：競爭者網站分析

⭐ 1-5 分：（直接回覆數字）
💬 建議：（選填）

你的回饋會幫助我優化未來的推薦！
```

## 5. 與其他技能的關係

| 技能 | 關係 | 說明 |
|------|------|------|
| `general-workflow` | 上層整合 | 任務來時先用 `general-workflow` 判斷，复雜任務再進 `skill-composer` |
| `agent-orchestrator` | 依賴 | 超複雜任務（L4）需要多智能體協作時，調用 `agent-orchestrator` |
| `site-qa-checklist` | 常被推薦 | 所有涉及網站輸出的任務幾乎都會推薦 |
| `metacognitive-learner` | 共生 | 學習推薦效果不好的原因，更新技能組合知識庫 |

## 6. 技術實作考量

### 6.1 技能元資料庫
- 掃描 `~/.hermes/skills/*/SKILL.md` 建立索引
- 每個技能解析：`name`, `description`, `trigger keywords`, `required_tools`
- 儲存格式：`~/.hermes/skills/_meta/skill-index.json`（定期更新）

### 6.2 組合儲存
- 使用 `~/.hermes/skills/_meta/combo-registry.jsonl`
- 每次新組合被使用時寫入一行（append-only log）
- 評價時更新對應行的 `evaluation` 欄位

### 6.3 觸發方式
- **被動觸發**：當 `general-workflow` 判定任務複雜度 >= L2 時呼叫
- **主動觸發**：當用戶說「推薦」「怎麼做」「要用什麼技能」
- **Cron 觸發**：每週執行一次「技能庫更新掃描」

### 6.4 LLM 作為推薦引擎
不硬編碼規則，而是用 LLM 理解任務本質並推薦：
- 輸入：任務描述 + 技能索引（壓縮版）
- 輸出：推薦組合 + 理由 + 預估效果分數
- 每次成功/失敗的使用經驗作為 few-shot examples

## 7. 第一版 MVP 範圍（Scope）

### 7.1 MVP 功能
1. **技能掃描器**：讀取 `~/.hermes/skills/` 建立元資料（名稱、描述、觸發關鍵字）
2. **簡單推薦**：根據任務描述 + 關鍵字匹配推薦技能（不超過 4 個）
3. **組合註冊**：記錄每次使用的組合（純文字，無評價系統）
4. **手動評價**：用戶可輸入 1-5 分評價

### 7.2 MVP 不包含
- 自動化評分追蹤
- 網路評價抓取
- 多智能體協作調度
- 即時分數排行系統

### 7.3 成功標準
- 用戶說「做 XXX」時，能給出合理的技能組合建議
- 组合建議有具體理由，不是隨機挑選
- 記錄的組合可在日後被檢索

## 8. 長期願景

最終目標：建立一個「技能組合的大腦」——
- 不只是「工具推薦」，而是「工作流設計」
- 每個複雜任務都被分解為可复用的「技能組合模板」
- 隨著使用次數增加，推薦越來越精準
- 形成赫米斯自己的「最佳實踐知識庫」

---

## 9. 待解決的問題

1. **技能能力邊界如何定義？** 目前只能從 `description` 萃取，未來需要更精準的「技能能力文件」
2. **組合爆炸問題**：100 個技能可能產生指數級的組合數量，如何有效篩選？
3. **評價主觀性**：用戶的 5 分和網路的 5 分是否等價？
4. **與 `agent-orchestrator` 的邊界**：什麼時候該用人類專家模式、什麼時候用多智能體？
